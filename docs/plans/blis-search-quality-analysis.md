# BLIS Router Search Quality Analysis — 260306_50i_real

## Executive Summary

All 4 search algorithms converged to ~30-33% improvement. This is **not a coincidence** — it reflects a real structural ceiling imposed by the evaluation setup. The algorithms found genuinely useful routing principles, but they also overfit to the specific workload configurations. Below is a detailed breakdown.

---

## Q1: Why is improvement stuck at ~30% for all algorithms?

### Short answer: The ceiling is real, but it's the ceiling of THIS evaluation, not of routing optimization in general.

### The conflicting-workload trap

The 3 workloads demand fundamentally different routing strategies:

*Source: `comparison_table.csv` generated by `compare_results.py`*

| Workload | Optimal strategy | Baseline E2E (ms) | Best evolved E2E (ms) | Per-workload improvement |
|---|---|---|---|---|
| `cache_warmup` | Load-first (prefix creates imbalance) | 6721.80 | 2465.94 (EvoX) | **-63.3%** |
| `load_spikes` | Load-balance (prefix is harmful) | 6126.69 | 6030.57 (AdaEvolve) | -1.6% |
| `multiturn` | Prefix-affinity (sessions need stickiness) | 1658.67 | 420.76 (GEPA) | **-74.6%** |

**The big story**: cache_warmup (-63%) and multiturn (-75%) see massive improvements. **load_spikes is essentially flat** — all frameworks are within ±5% of baseline, meaning the baseline's load-balance scorer already handles bursty traffic well.

**Why cache_warmup improves so much**: The baseline's 50/50 prefix-affinity + load-balance weighting creates load imbalance when 3 prefix groups map to 4 instances (one instance is underutilized). All evolved algorithms add load-relative penalties that actively shift traffic to the idle instance.

**Why multiturn improves so much**: The baseline's equal weighting causes session bouncing — when a cached instance is slightly busier, load-balance pulls traffic away, causing cache misses on 4096-token prefixes that cost hundreds of ms (especially on 14B). All evolved algorithms add explicit session affinity boosts and reduce load sensitivity for session traffic.

### Why all algorithms converge to the same range

1. **14B dominates the score**: With 14B avg latency ~8800ms vs 7B ~873ms, the combined score is ~90% determined by 14B performance. All algorithms that learn to handle 14B queueing well land in the same ballpark.

2. **Low-hanging fruit is the same for everyone**: The baseline's weakest points — ignoring KV pressure, no session affinity boost, no SLO-class differentiation — are obvious to any LLM-guided search. All 4 algorithms independently discover the same 4 patterns:
   - KV utilization penalty above 70-85%
   - Session/cache affinity boost
   - Relative load penalties (vs cluster min, not absolute)
   - SLO-class differentiation (realtime = stricter)

3. **Diminishing returns from the scoring function**: `score = -0.5 * avg_e2e - 0.5 * p95` averages across 6 runs (3 workloads × 2 models). Improving one workload's strategy by 5% gets diluted to <1% of the combined score, below the noise floor of the evaluation.

### Is ~33% the actual ceiling?

**Probably not.** Evidence that more improvement is possible:

- `cache_warmup` shows only -18% improvement — the smallest gain. Algorithms haven't fully learned to deprioritize prefix-affinity for short-input, high-rate traffic on this workload
- All algorithms plateau due to search exhaustion (50 iterations, 2-84% build error rates eating into the budget), not because they found the optimum
- Most snapshot signals are stale (5s refresh): QueueDepth, BatchSize, KVUtilization, FreeKVBlocks are all Periodic. Only InFlightRequests is fresh (cluster-level, updated every routing call). An algorithm that leans heavily on InFlightRequests and treats other signals as coarse hints could avoid oscillation
- No algorithm implements true probabilistic routing (random tiebreaking, exploration) — EvoX's power-of-two-choices is closest but its implementation is deterministic

**The ceiling within this evaluation setup is probably ~35-40%.** The main constraint is that the 3 workloads demand conflicting strategies, and any single routing function must compromise. A per-workload-adaptive router (detecting workload regime at runtime) could push further but is architecturally different from what the EVOLVE-BLOCK allows.

## Q2: Which algorithms are realistic vs. overfit?

### Assessment Framework

For each algorithm I check:
- **Principled mechanisms**: Does the approach have theoretical backing?
- **Magic numbers**: How many hand-tuned constants? Are they suspiciously precise?
- **Workload-specific fitting**: Would the algorithm break under different traffic patterns?
- **Robustness**: Does it degrade gracefully or cliff-edge?

---

### EvoX — Most production-ready (+33.1%)

**Principled mechanisms**:
- Hard filtering (remove overloaded instances) — well-established in load balancing
- Power-of-two-choices sampling — proven technique for reducing herd effects (Mitzenmacher 2001)
- Session affinity with cache hit threshold — reasonable gate

**Magic numbers** (7 constants):
- `loadThreshold = minLoad + 3` (realtime: +2) — reasonable but tuned
- `kvOk = 0.85` — reasonable KV cutoff
- `loadPenalty = 0.9 / (1 + 0.5*load)` — 0.9 and 0.5 are fit
- `CacheHitRate > 0.3` gate, `0.6` boost — tuned
- `FreeKVBlocks > 2000`, `1.2` boost — **red flag**: 2000 blocks is specific to 4×H100 cluster
- `BatchSize > 0` → `1.1` boost for batch — benign

**Overfitting risk: MODERATE.** The filtering + power-of-2 structure is sound. The FreeKVBlocks threshold would break on different hardware. The step-size sampling (lines 202-209) is a clunky approximation of power-of-two-choices that only works by accident with small candidate sets.

**Verdict**: Strip out the FreeKVBlocks check and batch-size bonus, keep the filter→score→sample pipeline. This is the most architecturally transferable.

---

### GEPA — Best score but fragile (+33.2%)

**Principled mechanisms**:
- Adaptive load penalty based on cluster pressure — novel and sound
- Separate queue depth vs batch size penalties — correctly distinguishes queuing delay from processing delay
- Cubic load penalty — aggressive but prevents hot spots

**Magic numbers** (12+ constants):
- `basePenalty` = 0.58 / 0.53 / 0.48 at `clusterPressure` thresholds 4.5 / 7.0 — **highly suspicious**: these thresholds are precisely tuned to the load levels in the 3 workloads
- `excessRatio` threshold at `1.12 * avgLoad` — why 1.12?
- Queue penalty 0.22, batch penalty 0.12 — arbitrary
- KV cutoffs at 0.7 and 0.8 with multipliers 0.6 and 0.3 — hard cliffs
- Min-load bonuses: 0.15 / 0.12 / 0.08 at load levels 0 / <3 / <6 — **clearly overfit** to specific load distributions

**Overfitting risk: HIGH.** The `clusterPressure` thresholds (4.5, 7.0), the `excessRatio` gate (1.12), and the min-load bonus tiers are all suspiciously precise. Under different workloads (e.g., rate=500, cluster size=8), these constants would be wrong.

**The KV penalty is a hard cliff** (0.3x multiplier at >0.8 utilization). In a real system with 5-second-stale KV signals, this cliff would cause oscillation: an instance crosses 0.8 → gets 0.3x → traffic shifts away → utilization drops → gets full traffic → crosses 0.8 again.

**Verdict**: The idea of adaptive cluster-pressure penalties is excellent and worth keeping. The specific implementation would need to be re-parameterized for any different deployment.

---

### AdaEvolve — Interesting ideas, overfitting risk (+31.3%)

**Principled mechanisms**:
- Input-length-aware load weighting (longer inputs → more load-sensitive) — makes physical sense: longer inputs occupy instances longer
- SLO-class multiplier — correct directionally
- Smooth exponential KV penalty using Padé approximant — sophisticated, avoids cliffs

**Magic numbers** (8 constants):
- `lw = 0.05 + 0.0004 * inputLen` — **directly fit** to the workload input distributions (mean 80-600 tokens). At inputLen=250 (typical), lw=0.15. At inputLen=1000 (capped), lw=0.45. These make no sense for inputs >1000 tokens.
- SLO multipliers: realtime=2.0, batch=0.5 — reasonable but arbitrary
- Cache affinity boost: `1.0 + 0.4 * CacheHitRate` — 0.4 is fit
- Excess load penalty: `1.0 / (1.0 + 0.2 * excess)` — 0.2 is fit
- KV threshold 0.7, penalty cap 0.3 — reasonable

**Overfitting risk: MODERATE-HIGH.** The linear input-length formula is the main concern — it's directly calibrated to workload input token distributions. The `inputLen > 1000` cap means this algorithm would treat a 2000-token input identically to a 1000-token input, which is wrong in production.

**Verdict**: The concept of input-length-aware routing is genuinely valuable and not something the baseline does. But the specific linear formula needs to be replaced with something more general (e.g., logarithmic scaling).

---

### OpenEvolve — Most elegant, least fit (+30.5%)

**Principled mechanisms**:
- Completely replaces the scoring: `score = prefixW * original + (1-prefixW) * loadScore` — this is a clean blend of the two objectives
- Adaptive prefix weight: reduces under load — correctly captures the insight that cache affinity should yield to load balance under congestion
- Quadratic+linear load: `1/(1 + load²*0.04 + load*0.25)` — smooth, no cliffs

**Magic numbers** (7 constants):
- prefixW = 0.4 (normal), 0.6 (session) — reasonable
- Load threshold for prefix reduction: `load > 10` → `prefixW *= 0.25` — **fit to specific load levels**
- Load function coefficients: 0.04, 0.25 — fit
- Excess penalty: 0.45 — fit
- KV threshold: 0.75, penalty slope 4.0 — reasonable

**Overfitting risk: LOW-MODERATE.** Fewest magic numbers, cleanest architecture. The `load > 10` threshold is the only really suspicious constant. The quadratic load function degrades gracefully.

**Verdict**: Despite having the worst score, this is the most generalizable architecture. The blend-weight approach is clean and interpretable. The `load > 10` threshold should be replaced with a percentile-based threshold (e.g., load > 2 * avgLoad).

---

### Ranking by production-readiness (not by benchmark score)

1. **EvoX** — Filter→Score→Sample pipeline is robust and well-understood
2. **OpenEvolve** — Clean adaptive blending, most generalizable
3. **AdaEvolve** — Good ideas (input-length-aware) but needs reparameterization
4. **GEPA** — Best benchmark score but most fragile in production

## Q3: What causes overfitting and how to fix it?

### Root causes of overfitting in this evaluation

| Root cause | Why it enables overfitting | Impact |
|---|---|---|
| **Only 3 workloads** | Algorithms can memorize the specific traffic patterns (rates, prefix groups, SLO mix) | All magic-number thresholds are calibrated to these 3 scenarios |
| **Fixed seed=42** | Every evaluation produces identical request sequences | Algorithms can fit to specific request ordering and timing |
| **Fixed cluster size (4 instances)** | Thresholds like `minLoad+3` or `FreeKVBlocks>2000` are specific to 4 instances | Won't transfer to 2, 8, or 16 instance clusters |
| **Only 2 models** | 14B dominates scoring, so algorithms optimize for 14B load patterns | Won't transfer to 70B, MoE, or mixed-model clusters |
| **No hold-out validation** | Training and test set are identical | No way to detect overfitting |
| **Equal avg/P95 weighting** | `score = -0.5*avg - 0.5*p95` may not reflect real priorities | Algorithms optimize for this specific tradeoff |
| **Stale KV signal** | All algorithms try to use KVUtilization despite 5s staleness | In production, KV staleness varies; hardcoded thresholds break |

### Concrete suggestions to reduce overfitting

#### A. Diversify workloads (highest impact)

Add workloads that stress different regimes:

```
# Suggested new workloads:
workload_v2_high_rate.yaml       # rate=1000+, all prefixed, tests pure load balancing under pressure
workload_v2_small_cluster.yaml   # same workloads but NUM_INSTANCES=2
workload_v2_large_cluster.yaml   # NUM_INSTANCES=8 or 16
workload_v2_no_prefix.yaml       # zero prefix traffic, pure load balancing
workload_v2_uniform_load.yaml    # steady poisson, no bursts, tests if algorithm adds unnecessary complexity
workload_v2_mixed_slo.yaml       # 50% realtime + 50% batch, extreme SLO differentiation
```

The evaluator already supports multiple workloads — just add more YAML files to the `WORKLOADS` list.

#### B. Randomize evaluation seeds (medium impact)

Instead of fixed seed=42, run each candidate with 3-5 different seeds and average:

```python
# In evaluator.py, change:
SIM_SEED = os.environ.get("BLIS_SEED", "42")

# To something like:
SIM_SEEDS = ["42", "123", "456"]  # evaluate each, average scores
```

This prevents algorithms from fitting to specific request orderings. Cost: 3x evaluation time (from ~30s to ~90s per iteration).

#### C. Randomize cluster size (medium impact)

Vary `BLIS_NUM_INSTANCES` across evaluations (e.g., randomly pick from {2, 4, 6, 8}). This prevents algorithms from hardcoding thresholds tuned to 4 instances.

#### D. Hold-out validation (high impact, post-search)

After search completes, evaluate the best program on a held-out set of workloads that weren't used during search. This doesn't prevent overfitting during search but detects it afterward.

Suggested held-out workloads:
- Different request rates (50 req/s, 500 req/s)
- Different prefix configurations (1 prefix group, 10 prefix groups)
- Different model (Mixtral 8x7B)
- Different cluster sizes

#### E. Complexity penalty in scoring (low-medium impact)

Add a term that penalizes the size of the EVOLVE block:

```python
# Lines of code in evolved block
loc_penalty = max(0, (evolved_loc - baseline_loc) * 0.001)
score = -0.5 * avg_e2e - 0.5 * avg_p95 - loc_penalty
```

This discourages algorithms from adding many special-case branches that are likely overfit.

#### F. Workload perturbation during search (high impact)

The most effective anti-overfitting technique: for each evaluation, randomly perturb the workload parameters:

```python
# Before each simulation run, jitter the workload:
# - rate: ±20%
# - num_requests: ±10%
# - prefix_length: ±15%
# - input distribution mean: ±20%
```

This forces evolved algorithms to be robust to workload variation. Implementation: add a pre-processing step in the evaluator that copies and perturbs the YAML before passing to the simulator.

## Algorithm-Specific Observations

### Why is cache_warmup improvement so large (-63%)?

The baseline's 50/50 prefix-affinity + load-balance weighting creates a natural imbalance: 3 prefix groups map to 3 of 4 instances, leaving the 4th underutilized. The baseline's `1/(1+load)` scorer is too gentle — it doesn't penalize the loaded instances enough to redirect traffic to the idle one. All evolved algorithms add **relative load penalties** (comparing each instance to cluster minimum) which aggressively shift traffic to underutilized instances. The 14B model amplifies this: with ~10x higher per-request latency, load imbalance creates much larger queueing effects.

### Why is multiturn improvement so large (-75%)?

The baseline's equal weighting causes session bouncing. Prefix-affinity correctly routes sessions to cached instances, but load-balance fights it: when a cached instance is slightly busier, load-balance overrides affinity, causing cache misses on 4096-token prefixes (72ms+ per miss on 7B, hundreds of ms on 14B). All evolved algorithms add **explicit session affinity boosts** (checking `req.SessionID` and `snap.CacheHitRate`) and **reduce load sensitivity for session traffic**, which the baseline's rigid 50/50 weighting can't do.

### Why is load_spikes essentially flat (±5%)?

The baseline's load-balance scorer already handles the heavy-hitter prefix pattern well. With 50% of traffic on one prefix group, the load-balance component already avoids piling everything onto the cached instance. The evolved algorithms' additional load penalties (relative to min, KV-aware, etc.) don't add much value because the baseline is already near-optimal for this workload. Some frameworks (GEPA, EvoX) even slightly regress on load_spikes while improving the other two workloads — a sign that their session affinity boosts cause minor load imbalance during bursts.

---

## The real ceiling question: What would break through 35%?

1. **Workload-regime detection at runtime**: If the router could detect "this looks like bursty traffic" vs "this looks like multi-turn sessions" and switch strategies, it could do much better. But the EVOLVE-BLOCK doesn't have access to request history — it only sees current request + snapshots.

2. **Leaning on InFlightRequests**: QueueDepth, BatchSize, and KVUtilization are ALL stale (5s refresh). Only InFlightRequests is fresh (updated every routing call). An algorithm that uses InFlightRequests as the primary differentiator between instances and treats stale signals as coarse background context would avoid the oscillation that stale-signal-based penalties cause. None of the evolved algorithms make this distinction — they all use EffectiveLoad() (partially stale) or stale fields directly.

3. **Probabilistic routing with exploration**: Instead of always routing to the highest-scoring instance, occasionally routing to the 2nd-best creates better load distribution. EvoX's power-of-two-choices is the closest to this but its implementation is flawed (deterministic stepping through candidates, not actual random sampling).

4. **Time-aware routing**: If the algorithm could distinguish "I'm in a burst" (many recent requests) from "traffic is calm" and adjust aggressively, it would handle load_spikes much better. This would require adding a request counter or timestamp to the EVOLVE-BLOCK context.

## Transfer to llm-d: Strategy

### The end goal

Pick one algorithm, port it to llm-d's Go routing, test against same/similar workloads in the real system to demonstrate improvement. This changes what "overfitting" means.

### What overfitting matters and what doesn't

| Concern | Matters for transfer? | Why |
|---|---|---|
| Magic numbers tuned to specific load levels (4.5, 7.0, minLoad+3) | **Yes** | Real system load distributions will differ from simulator |
| Fitting to 3 specific workload structures (prefix groups, SLO mix) | **Less** if testing same workloads | If llm-d tests use similar workloads, this is fine |
| Fitting to simulator artifacts (exact snapshot timing, DES event ordering) | **Yes, biggest risk** | Real vLLM/llm-d has different scheduling, batching, network delays |
| Fitting to 4-instance cluster | **Moderate** | Must work at different cluster sizes in production |
| Fitting to seed=42 request ordering | **Low** if workloads are long enough | 9K-18K requests average out most ordering effects |

### The real risk: simulator vs. reality gap

The algorithms are optimized against a Go discrete-event simulator. The real system differs in:
- **Batching dynamics**: Real vLLM continuous batching doesn't match the simulator's batch formation model exactly
- **Network delays**: Real systems have variable network latency between router and instances
- **Snapshot delivery**: Real Prometheus scrape has jitter; simulator has exact 5s intervals
- **Preemption**: Real systems may preempt requests; simulator may not model this

These differences mean an algorithm that gets 33% improvement in simulation might get 15-25% in the real system — still great, but expect degradation.

### Recommendation: what to transfer

**Pick EvoX's architecture, simplify the constants.**

Reasons:
1. The filter→score→sample pipeline is the most robust to environment change — hard filtering prevents catastrophic decisions even if thresholds are slightly wrong, and power-of-two-choices handles herd effects naturally
2. It doesn't rely heavily on stale signals (KV threshold is just a filter, not a continuous penalty)
3. The architecture has a known theoretical basis (power-of-two-choices literature)

**Simplification for transfer:**
```
Stage 1: Compute base scores from existing scorers (keep this unchanged)
Stage 2: Filter out instances where EffectiveLoad > minLoad + K (K=3 default, K=2 for realtime)
Stage 3: Boost score for session cache hits
Stage 4: Pick best from 2-3 random candidates (true power-of-two-choices)
```

Drop the FreeKVBlocks check (hardware-specific) and batch-size bonus (minor effect). Replace deterministic stepping with actual random sampling.

### What to do about 14B dominating the score

Two options depending on your goal:

**Option A: Keep as-is (recommended for the paper).** If you're showing "routing optimization helps heavier models more" — that's a great story. In production, people run 70B and MoE models that are even heavier. The 14B domination reflects a real insight: routing optimization has more leverage on expensive models.

**Option B: Normalize per-model.** If you want algorithms that are equally good for both small and large models, change the evaluator to compute per-model scores separately, then average:
```python
# Instead of averaging all 6 latencies together:
per_model_scores = []
for model in models:
    model_avg = average(model_latencies)
    model_p95 = average(model_tail_latencies)
    per_model_scores.append(-0.5 * model_avg - 0.5 * model_p95)
score = average(per_model_scores)
```
This gives 7B and 14B equal weight. But it may produce algorithms that compromise on 14B to help 7B, which is worse for the transfer story.

### Workload strategy for the paper

Since you're testing against same/similar workloads in llm-d:

1. **Keep the 3 workloads for search** — they cover the key routing tensions (cache imbalance, burst absorption, session stickiness)
2. **Add 1-2 seeds during search** (seeds 42+123) — cheap insurance against ordering artifacts, doubles eval time but catches fragile algorithms
3. **Create 1-2 held-out workloads for llm-d validation** — slight variations of the 3 (different rates, different prefix sizes) to show the algorithm generalizes beyond the exact training setup
4. **Don't add workload perturbation** — adds complexity and slows search. The transfer test in llm-d IS the real validation
